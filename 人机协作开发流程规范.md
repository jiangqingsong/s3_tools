# 人机协作开发流程规范

本文档定义了需求提出方（用户）与 AI（Claude Code）之间的标准协作流程，目标是保证开发准确性和沟通效率。

## 流程总览

```
需求输入 → 方案设计 → 开发计划 → 分步实现 → 验证交付
   │           │           │           │           │
   │           │           │           │           └─ 最终验收
   │           │           │           └─ 逐 Phase 编码，每步验证
   │           │           └─ 拆成有序、可验证的实施步骤
   │           └─ 翻译为技术方案，对齐架构和接口
   └─ 说清要什么，追问边界
```

## 各阶段详解

### Phase 1：需求输入
- **目标**：明确要做什么，边界在哪里
- **用户职责**：描述需求场景、期望的功能范围
- **AI 职责**：追问边界条件、澄清模糊点、识别潜在风险
- **产物**：无文档，对话中达成共识
- **完成标准**：双方对"要做什么"认知一致

### Phase 2：方案设计
- **目标**：将需求翻译为具体的技术方案
- **AI 职责**：输出 `DESIGN.md`，包含技术选型、架构设计、接口定义、数据结构、异常处理策略
- **用户职责**：审阅方案，纠正方向偏差，确认决策点
- **产物**：`DESIGN.md`
- **完成标准**：用户明确表示"方案确认，可以出开发计划"
- **关键原则**：这一阶段只改文档不写代码，修改成本为零

### Phase 3：开发计划
- **目标**：将方案拆解为有序、可验证的实施步骤
- **AI 职责**：输出 `DEVELOPMENT_PLAN.md`，按依赖关系排列任务，每个任务标注具体的产出物和验证方式
- **用户职责**：确认开发节奏和优先级
- **产物**：`DEVELOPMENT_PLAN.md`
- **完成标准**：用户确认"按计划开始实施"
- **关键原则**：每个任务必须可验证——明确用什么命令、期望什么结果

### Phase 4：分步实现
- **目标**：按开发计划逐 Phase 编码
- **AI 职责**：按计划顺序编码，每完成一个任务自测验证，更新进度状态
- **用户职责**：必要时验收关键节点
- **产物**：可运行的代码
- **关键原则**：
  - 严格按 Phase 顺序执行，不跳步
  - 每步验证通过后再进入下一步
  - 遇到方案未覆盖的细节，先讨论再动手

### Phase 5：交付验收
- **目标**：最终确认所有功能符合预期
- **AI 职责**：汇总完成情况，标记遗留项
- **用户职责**：验收功能，决定是否进入下一轮迭代
- **产物**：可运行的服务 / 功能模块

## 沟通规范

### 引用约定
对话中引用文档内容时，使用精确引用：
- 文档：`DESIGN.md` 或 `DEVELOPMENT_PLAN.md`
- 章节：`第 3.2 节` 或 `Phase 2`
- 任务：`任务 2.3` 或 `Phase 3 第 3.4 步`

### 状态同步
开发计划中的进度表是双方的"事实来源"。AI 在每步完成后更新状态，用户随时通过进度表了解整体情况。

### 遇分歧时
当编码过程中发现方案不合理时：
1. AI 指出问题，暂停编码
2. 回退到方案文档讨论
3. 更新 `DESIGN.md` 后同步更新 `DEVELOPMENT_PLAN.md`
4. 继续编码

## 文档体系

| 文档 | 定位 | 更新时机 |
|------|------|----------|
| `DESIGN.md` | 技术方案"做成什么样"（接口、架构、数据） | 方案设计阶段，少量在实现阶段修正 |
| `DEVELOPMENT_PLAN.md` | 实施路线"按什么顺序做"（步骤、验证） | 开发计划阶段，实现阶段持续更新进度 |
| 代码 | 实际实现 | 分步实现阶段 |

核心原则：**方案驱动开发，计划追踪进度，文档先于代码。**
